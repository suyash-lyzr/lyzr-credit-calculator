# Lyzr Credit Calculator

## Overview
The Lyzr Credit Calculator is an interactive application built with Next.js 15, TypeScript, and Anthropic Claude. Its primary purpose is to help users understand the cost and ROI of building AI agents on the Lyzr platform. It features a Claude-like artifact experience and utilizes a transparent Agent Run pricing model. The project aims to provide clear credit and ROI calculations based on user-defined use cases, facilitating informed decisions for adopting Lyzr's AI agent solutions.

## User Preferences
- Layout: Landing page → 40% chat, 60% calculation results
- Uses own Anthropic API key
- Terminology: "use case" instead of "automation"

## System Architecture
The application uses Next.js 15.5.0 with the App Router, TypeScript 5, and Anthropic Claude Opus 4.7 (`claude-opus-4-7`, using the user's API key). The temperature parameter is intentionally omitted — it is deprecated on this model. Styling is handled by Tailwind CSS 4 and shadcn/ui. Diagrams are rendered with Mermaid.js, and markdown support is provided by react-markdown.

The user interface transitions from a simple landing page to a 40-60 split layout after the initial user input. The AI assesses complexity (LOW/MEDIUM/HIGH) and generates four artifacts: an agent architecture diagram, a credit cost calculation showing Lyzr agent-run cost and LLM pass-through cost separately, an ROI comparison against human labor costs, and a review/validation pass.

The core pricing model is the **complexity-tier model** (revamped 2026). Cost = Lyzr Platform Cost + LLM Cost. Platform cost depends on two things: **number of runs × complexity tier**, where a run = one execution request (one chat message, one pipeline fire, one schedule tick). Tiers: **Simple/Single Agent** (flat $0.06 SaaS / $0.03 On-Prem), **Intermediate/Manager** (banded by sub-agents executed: $0.18/$0.36/$0.54 SaaS), **Complex/Superflow** (banded by nodes executed: $0.30/$0.60/$0.90 SaaS, then incremental >30), **Voice** (per minute $0.09/$0.06). LLM cost is pass-through at provider rates with no markup (and $0 on the Lyzr bill if the customer brings their own model). A use case decomposes into one or more **workloads**, each assigned the cheapest orchestration tier that does the job (mixtures are normal). The system includes tools for `generate_architecture` (decompose into tiered workloads + diagram), `calculate_credits` (the model supplies the DESIGN; the server's `src/lib/pricing.ts` engine computes all costs deterministically), `web_search`, `calculate_roi`, and `review_and_validate`.

**Authoritative pricing spec: `context/PRICING-METHODOLOGY.md`** (plus supporting `context/*.md` — orchestration rubric, superflow/agent capabilities, verified LLM model rates). The deterministic rate card + cost functions live in `src/lib/pricing.ts`. NOTE: passages further below describing the old flat-rate "agent run = reasoning step" volume waterfall are superseded by the complexity-tier model.

LLM cost estimation uses a 4-step framework codified in the system prompt: (A) classify each agent by task type, context size, volume, and quality bar; (B) pick the cheapest model that clears the quality bar from the May 2026 lineup (OpenAI GPT-5.4 Nano / GPT-5.4 / o4-mini / o3 / o3-pro, Anthropic Claude Haiku 4.5 / Sonnet 4.6 / Opus 4.7, Google Gemini 3.1 Flash-Lite / Gemini 3 Flash / Gemini 3.1 Pro with 2M context); (C) mix models across agents in multi-agent architectures (cheap routers, mid-tier workers, strong orchestrators); (D) estimate tokens per run by task type. Provider rates are baked into the prompt and refreshed for May 2026.

Volume model treats user-stated volume as the bottom-of-funnel successful output, scoped to one team / type / geography (typically 3-10× understated vs true enterprise workload). The model decomposes the true workload into: (1) funnel multiplier on volume (`unit_volume × funnel_multiplier = effective_unit_volume`) — regulated enterprise floor 2.5×, going up to 4-5× for legal/financial/healthcare across all teams; (2) per-step `iteration_multiplier` on each `runs_breakdown` row capturing rework/revision/approval loops — regulated defaults are extraction 2-4×, validation 3-5×, drafting 4-7×, approval gate 5-8×; (3) single outer `iteration_buffer_pct` with a procurement-defensible 50% anchor for any regulated/enterprise/first-deployment workflow (range 40-60%, going to 60-80% if Year-1 onboarding is bundled), covering continuous copilot/KB/audit ops (~15-25%), ramp-up calibration, surge volume, novel inputs, retries, and exception handling; (4) Year-1 onboarding for regulated rollouts (playbook ingestion, corpus indexing, library training, UAT cycles — typically 100K-200K one-time runs) emitted as its own workload row in `rows[]` rather than buried in the buffer. Each `runs_breakdown` step is tagged with a `phase` (discovery / validation / processing / review / delivery / monitoring) and rolled up in `phase_breakdown[]`. The defaults library covers document processing (structured + regulated), marketing test-and-learn, decision workflows (regulated), customer support, real-time decisioning, research/analyst, code/engineering, and sales workflows. The model is instructed to never enumerate "KB lookups / tool calls / sub-agent calls / memory / guardrails / security are free inside an agent run" in customer-facing chat or artifacts — the artifact's intro line "Lyzr charges only for agent runs. LLM costs pass through at provider rates — no markup." is the only line shown.

Design language: warm earthy palette — cream/beige background `hsl(36, 33%, 94%)` with a faint brown grid texture, deep brown primary `hsl(25, 62%, 25%)`, lighter cream input surfaces via `--input-bg` token (`hsl(36, 50%, 99%)`). Typography pairs Playfair Display (`--font-serif`, all `h1`–`h6`) with DM Sans (`--font-sans`, body). Radius scale: `0.5rem` default, `1rem`/`1.5rem`/`2rem` for `lg`/`xl`/`2xl`. CTAs use solid `bg-primary` (the gradient utility `bg-gradient-primary`/`text-gradient-primary` exists but is reserved for accent use, not CTA fills). Mermaid architecture diagrams use the same brown palette (`#673F1B` nodes, `#3D2510` borders/lines).

The application features a streaming API (`/api/chat`) for real-time text output and tool execution, using Server-Sent Events (SSE).

Authentication uses passwordless email OTP (6-digit code, SHA-256 hashed, 10-min TTL, single active code per email enforced via row replacement, 5 attempts per code). OTP request and verify endpoints are rate-limited per email and per IP via an in-memory sliding-window limiter (`src/lib/rate-limit.ts`). Sessions are stored in the `sessions` table (32-byte token, 30-day TTL) and tracked via an httpOnly `lyzr_session` cookie (sameSite=lax, secure in production). `middleware.ts` redirects unauthenticated browser requests to `/login` and returns 401 for API requests, but every protected API route additionally calls `getCurrentUser()` server-side (defense in depth — middleware only checks cookie presence). SMTP is configured via `SMTP_HOST` / `SMTP_PORT` / `SMTP_USER` / `SMTP_PASS` / `SMTP_FROM` env vars and uses nodemailer (port 465 → SSL, otherwise STARTTLS).

Chat history and saved templates are persisted per-user in PostgreSQL (`chat_sessions`, `templates` with `userId` FK, cascade delete). The credit-calculator client loads sessions from `/api/chat-sessions` on mount, debounces saves at 600ms, flushes pending writes on session switch and `beforeunload`, and keeps an in-memory artifacts cache in sync with live state so re-selecting a session never shows stale data. localStorage-based session storage has been removed.
